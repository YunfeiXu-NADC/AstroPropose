# AstroPropose 用户管理设计

日期：2026-04-21

## 1. 背景

当前 AstroPropose 已具备基础认证与角色判断能力：

- 用户可注册、登录，并通过 `/api/auth/me` 获取当前身份信息
- 后端已存在 `User`、`Role`、`roles_users` 多对多模型
- 系统已有基于角色的访问控制，例如 `Admin`、`Panel Chair`、`Instrument Scheduler`
- 工作流流转已支持 `allowed_roles`

但系统尚缺少完整的用户管理能力，主要表现为：

- 没有管理员用户列表与编辑界面
- 没有管理员创建用户、分配角色、禁用用户、重置密码的入口
- 没有普通用户修改自己密码的能力
- 没有角色管理页面，角色主要依赖种子数据或直接改数据库

本设计用于补齐第一版可用的用户与角色管理闭环。

## 2. 目标

本次实现目标：

- 管理员查看全部用户
- 管理员创建用户
- 管理员编辑用户邮箱
- 管理员分配和撤销用户角色
- 管理员禁用和启用用户
- 管理员重置用户密码
- 普通用户修改自己的密码
- 管理员创建、重命名、删除自定义角色
- 让自定义角色可用于工作流 `allowed_roles`

## 3. 非目标

本次不做以下内容：

- 不实现完整权限点模型，不引入 `Permission` 表
- 不将自定义角色直接映射为系统后台菜单权限
- 不允许修改用户名
- 不实现忘记密码、邮件找回密码、短信验证
- 不实现审计日志中心
- 不实现批量导入导出用户
- 不实现角色层级继承

## 4. 设计原则

- 保持现有 `User` 与 `Role` 多对多结构，避免大范围重构
- 区分“系统角色”和“自定义角色”
- 系统后台能力仍由内置系统角色控制，自定义角色主要服务于业务流转与用户分组
- 禁用用户而不是物理删除用户，保证 proposal 与历史记录可追溯
- 第一版优先完成最小闭环，避免把用户管理和完整 RBAC 一次性做重

## 5. 角色模型

角色分为两类：

- 系统角色：`Admin`、`Proposer`、`Panel Chair`、`Instrument Scheduler`
- 自定义角色：由管理员创建，用于扩展业务身份与工作流流转角色

系统角色规则：

- 可分配给用户
- 不允许删除
- 第一版不允许重命名
- 仍用于控制后台菜单与核心管理接口访问

自定义角色规则：

- 可由管理员新建、重命名、删除
- 可分配给用户
- 可在工作流 `allowed_roles` 中引用
- 默认不自动获得系统管理权限

## 6. 数据模型变更

### 6.1 Role

在 `Role` 模型中新增：

- `is_system: Boolean`

语义：

- `true` 表示系统角色
- `false` 表示自定义角色

迁移策略：

- 对已有角色进行初始化
- `Admin`、`Proposer`、`Panel Chair`、`Instrument Scheduler` 标记为 `true`
- 其他已有角色若存在，默认标记为 `false`

### 6.2 User

在 `User` 模型中新增：

- `is_active: Boolean`

语义：

- `true` 表示账号可登录
- `false` 表示账号已禁用

迁移策略：

- 默认值为 `true`
- 对历史用户统一填充为 `true`

### 6.3 兼容性

以下结构保持不变：

- `roles_users` 多对多关系
- `password_hash`
- `has_role()` 判断逻辑

这样可以最大化兼容现有后端接口和前端角色判断。

## 7. 权限规则

### 7.1 登录与认证

- 登录时除用户名密码校验外，增加 `is_active` 检查
- 若用户被禁用，登录返回 403，并给出明确提示
- 已登录用户若之后被管理员禁用，后续访问需要重新校验 `is_active`
- 因此 `token_required` 在解析 token 后，还需要校验用户存在且 `is_active = true`

### 7.2 管理员能力

只有具备 `Admin` 角色的用户可以访问：

- 用户列表
- 用户创建
- 用户编辑
- 用户禁用/启用
- 用户角色分配
- 管理员重置密码
- 角色管理接口

### 7.3 普通用户能力

普通用户仅可：

- 查看自己的当前身份信息
- 修改自己的密码

普通用户不能：

- 查看其他用户
- 修改其他用户角色
- 禁用其他用户
- 创建或删除角色

### 7.4 角色删除保护

删除角色前必须执行约束检查：

- 若 `is_system = true`，拒绝删除
- 若该角色已绑定任意用户，拒绝删除
- 若该角色已被任意工作流 transition 的 `allowed_roles` 引用，拒绝删除

只有同时满足以上保护条件后，才允许删除自定义角色。

## 8. API 设计

新增一个独立的管理员蓝图，例如 `/api/admin`，以避免把用户管理逻辑混入现有业务蓝图。

### 8.1 角色管理

#### `GET /api/admin/roles`

返回全部角色，字段包括：

- `id`
- `name`
- `is_system`
- `user_count`
- `workflow_reference_count`

#### `POST /api/admin/roles`

请求：

- `name`

规则：

- 名称不能为空
- 名称全局唯一
- 创建结果默认 `is_system = false`

#### `PATCH /api/admin/roles/<id>`

请求：

- `name`

规则：

- 系统角色不可改名
- 新名称不能为空
- 新名称不可与其他角色冲突

#### `DELETE /api/admin/roles/<id>`

规则：

- 系统角色不可删
- 已绑定用户不可删
- 已被工作流引用不可删

失败时返回明确原因，便于前端提示用户解除引用后再删除。

### 8.2 用户管理

#### `GET /api/admin/users`

返回全部用户列表，字段包括：

- `id`
- `username`
- `email`
- `is_active`
- `roles`

第一版不额外引入分页；若后续用户量上升，再补分页与搜索。

#### `POST /api/admin/users`

请求：

- `username`
- `email`
- `password`
- `role_ids`
- `is_active`，可选，默认 `true`

规则：

- 用户名唯一
- 邮箱唯一
- 用户名创建后不可改
- 默认至少给一个合理角色；若前端未选，建议默认赋予 `Proposer`

#### `PATCH /api/admin/users/<id>`

可更新字段：

- `email`
- `role_ids`
- `is_active`

不允许更新：

- `username`

规则：

- 邮箱唯一性校验
- 至少保留角色数组合法性
- 若尝试禁用当前登录管理员自己，第一版直接拒绝，避免把系统唯一管理员误锁死

#### `POST /api/admin/users/<id>/reset-password`

请求：

- `new_password`

规则：

- 仅管理员可调用
- 不要求旧密码
- 密码通过 `set_password()` 更新

### 8.3 用户自助修改密码

#### `POST /api/auth/change-password`

请求：

- `old_password`
- `new_password`

规则：

- 必须先登录
- 旧密码校验通过后才允许修改
- 修改成功后返回成功消息

## 9. 前端页面设计

### 9.1 导航

在管理员导航中新增：

- `Users`
- `Roles`

仅 `Admin` 可见。

### 9.2 用户管理页 `/admin/users`

页面结构：

- 用户列表表格
- “创建用户”按钮
- “编辑用户”按钮
- “重置密码”按钮
- “禁用/启用”按钮

表格列建议包括：

- 用户名
- 邮箱
- 角色列表
- 状态
- 操作

创建/编辑用户采用弹窗或侧边面板，字段包括：

- 用户名
- 邮箱
- 角色多选
- 启用状态

其中：

- 创建时显示密码输入框
- 编辑时不显示用户名可编辑项，用户名只读

### 9.3 角色管理页 `/admin/roles`

页面结构：

- 角色列表
- “新建角色”按钮
- “重命名”按钮
- “删除”按钮

角色列表字段包括：

- 角色名
- 类型（系统/自定义）
- 用户引用数
- 工作流引用数

交互规则：

- 系统角色不显示删除和重命名操作，或置灰
- 删除失败时直接展示“仍被用户或工作流引用”

### 9.4 用户修改密码入口

第一版不新建独立的个人资料中心。

在 Dashboard 或导航中增加一个简单入口：

- `Change Password`

进入一个轻量表单页面，或在 Dashboard 内弹窗处理：

- 当前密码
- 新密码
- 确认新密码

## 10. 与工作流的集成

工作流编辑器中的角色选项，当前存在部分写死或预置角色来源。第一版需要统一改为从后端角色接口动态获取，以便：

- 系统角色可见
- 自定义角色可见
- 新建角色后无需改代码即可用于 `allowed_roles`

这项改动只影响候选角色来源，不改变 `allowed_roles` 的存储结构，仍保持字符串角色名数组，以降低迁移复杂度。

## 11. 错误处理

前后端需要对以下错误给出清晰提示：

- 用户名已存在
- 邮箱已存在
- 旧密码错误
- 禁用账号禁止登录
- 非管理员访问管理员接口
- 系统角色不允许删除
- 角色仍被用户或工作流引用
- 自定义角色重名

错误响应风格保持与现有接口一致，返回 `message` 字段为主。

## 12. 测试策略

### 12.1 后端测试

新增测试覆盖以下行为：

- 禁用用户无法登录
- 已禁用用户即使持有旧 token，也无法继续通过鉴权
- 管理员可创建用户
- 管理员可编辑邮箱
- 管理员可调整角色
- 管理员可禁用和启用用户
- 管理员可重置密码
- 普通用户可修改自己的密码
- 普通用户旧密码错误时修改失败
- 非管理员访问 `/api/admin/*` 被拒绝
- 系统角色不可删除或改名
- 被引用角色不可删除
- 自定义角色可创建、改名、删除

### 12.2 前端验证

至少覆盖以下场景：

- `Admin` 用户可看到 `Users` 和 `Roles` 菜单
- 非管理员看不到相关入口
- 新建用户后列表正确刷新
- 编辑用户后角色显示正确
- 重置密码表单成功与失败提示正确
- 修改自己密码成功后可使用新密码重新登录

### 12.3 集成验证

需要确认以下联调结果：

- 新建自定义角色后，用户编辑页可分配该角色
- 新建自定义角色后，工作流角色候选中可见
- 禁用用户后，登录立即失效

## 13. 实施顺序

推荐顺序如下：

1. 数据模型变更：新增 `Role.is_system` 与 `User.is_active`
2. 编写后端测试，先写失败用例
3. 实现后端用户与角色管理接口
4. 实现 `/api/auth/change-password`
5. 前端补充 API 封装
6. 实现 `/admin/users`
7. 实现 `/admin/roles`
8. 在导航与 Dashboard 中增加修改密码入口
9. 将工作流角色候选改为动态来源
10. 跑后端测试、前端构建、联调验证
11. 部署到服务器验证

## 14. 风险与取舍

### 14.1 自定义角色不直接带系统权限

这是一个有意取舍。

原因：

- 当前代码中大量系统能力判断依赖硬编码角色名
- 若本次让自定义角色直接映射系统权限，会把任务扩大为完整权限模型重构

因此第一版只把自定义角色用于用户分组和业务流转。

### 14.2 删除角色受限

第一版不提供“强制删除并自动解绑所有引用”。

原因：

- 会带来不可预测的工作流后果
- 管理员应先显式解除用户绑定和工作流引用，再删除角色

### 14.3 不允许修改用户名

用户名通常已被用作登录标识。第一版禁止修改用户名，可以避免：

- 登录混乱
- 历史记录主体歧义
- 额外的唯一性和兼容性问题

## 15. 完成标准

当以下条件全部满足时，本需求可视为完成：

- 管理员可在前端查看、创建、编辑、禁用用户
- 管理员可重置密码
- 普通用户可修改自己的密码
- 管理员可创建、重命名、删除自定义角色
- 系统角色保护生效
- 被引用角色删除保护生效
- 工作流角色候选可见自定义角色
- 后端测试通过
- 前端构建通过
- 部署环境验证通过
